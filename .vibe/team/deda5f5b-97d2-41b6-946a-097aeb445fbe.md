### Architecture

Settings is reordered around Accounts, Privacy & Devices, Notifications, Appearance, and Data & Storage. Multi-account state is device-local and account-namespaced (session/config, keychain secrets, caches, sockets, push preferences); switching accounts tears down the old runtime before activating the new one. The server adds device-scoped sessions, linked-device records, one-use pairing requests containing only public material/wrapped key envelopes, and per-account notification preferences. iOS provides the avatar long-press switcher plus functional account/device, notification, QR/secret, storage, and appearance flows. Web keeps the existing React/Vite runtime but receives a cohesive auth/home/chat shell and the same persisted theme/wallpaper/gradient model.

### Contracts

- Auth: `Authorization: Bearer <device_session_token>`; tokens are stored server-side only as SHA-256 hashes. Legacy `users.login_token` remains accepted during migration.
- Devices: `GET /api/account/devices`, `DELETE /api/account/devices/:id`, `POST /api/account/devices/pairing`, `POST /api/account/devices/pairing/:code/approve`, `POST /api/account/devices/pairing/:code/claim`.
- Pairing request: `{code, requesterDeviceId, requesterName, platform, requesterPublicKey, expiresAt}`. Approval stores a device-targeted encrypted/wrapped key envelope; plaintext account secrets never transit or persist on the server.
- Claim response: `{account, device, sessionToken, wrappedKeyEnvelope}` and consumes the request atomically.
- Notifications: `GET/PUT /api/account/notification-preferences`; payload categories `privateChats`, `groupChats`, `channels`, `stories`, `reactions`, plus `allAccounts`, `inAppSounds`, `inAppVibrate`, `inAppPreview`, `namesOnLockScreen`.
- Appearance: shared semantic model `{mode, themeId, wallpaper, accent, bubbleGradient, textScale, messageCorners, animationsEnabled}` with account-scoped persistence and live preview.
- Cache: account-scoped byte counts for media, thumbnails, documents, audio, and temporary files; clear operations never remove keys or unsent messages.

### Decisions

- Device linking uses asymmetric key wrapping and short-lived single-use codes; no plaintext secret-key API.
- Device session rollout is backward-compatible with the current single user token.
- Theme and notification preferences are per account; a device may optionally show notifications for all locally registered accounts.
- Reference images inform hierarchy and interaction only; visuals remain Vibe-specific.

### Findings / blockers / next steps

- Current iOS persistence is global (`AppSessionConfig.current`, fixed ChatEngineStore keys, fixed Keychain names), so namespace migration is mandatory before switching accounts safely.
- Current server has a single `users.login_token` and push-token bundle, with no linked-device/session tables.
- Current web has separate landing and `/app` routes and functional chat behaviors that must be preserved during visual restructuring.
- Next: workers implement their exact paths; lead wires shared entry points, resolves integration gaps, and runs all builds/tests.
