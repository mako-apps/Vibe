### Architecture and contracts

The existing Phoenix API and React/Vite client remain in place. The existing uncommitted `account_devices`, `device_sessions`, `device_link_requests`, and `notification_preferences` migration is treated as shared foundation. Device linking transfers only a client-wrapped key envelope; plaintext secret keys never reach the server. iOS stores each account's credentials, key material, and cache namespace by user id and switches atomically by tearing down the active runtime before reconnecting. Notifications use category drill-down controls. Appearance uses one reusable theme contract across iOS and web: palette, incoming/outgoing gradients, wallpaper, dark variant, text scale, message corners, and animation preference.

API shapes:

- Account summary: `{id,user_id,username,name,profile_image,is_active}`.
- Device: `{id,name,platform,last_seen_at,is_current,revoked_at}`.
- Link: `POST /api/devices/link-requests`, approve with `wrapped_key_envelope`, then consume for session credentials.
- Notification preferences: categories `private_chats`, `group_chats`, `channels`, `stories`, `reactions`; flags `in_app_sounds`, `in_app_vibrate`, `in_app_preview`, `names_on_lock_screen`.
